# Dify 发布检查清单（Checklist）分析

> 对应 Dify 前端 "检查清单(N)" 弹窗 — 阻止带有配置问题的工作流发布

## 1. 实现位置

**100% 前端实现**，后端仅在发布时检查凭据和计费限制。

| 组件 | 文件 |
|------|------|
| 核心逻辑 | `web/app/components/workflow/hooks/use-checklist.ts` |
| UI 弹窗 | `web/app/components/workflow/header/checklist/index.tsx` |
| 单节点组 | `web/app/components/workflow/header/checklist/node-group.tsx` |
| 各节点验证 | `web/app/components/workflow/nodes/[type]/default.ts` 的 `checkValid()` |
| 变量提取 | `web/app/components/workflow/nodes/_base/components/variable/utils.ts` |
| 国际化 | `web/i18n/zh-Hans/workflow.json` |

## 2. 检查层次

检查清单对每个节点执行 **3 层检查**：

### 第 1 层：节点配置完整性 (`checkValid()`)

每种节点有独立的 `checkValid()` 函数，检查节点自身配置是否完整。
规则模式：`字段名 不能为空`

### 第 2 层：变量引用合法性（跨节点）

检查节点使用的所有 `value_selector` 引用是否指向有效的上游节点及其输出。
错误信息：`无效的变量。请选择一个已有的变量。`

### 第 3 层：连接合法性

检查节点是否连接到工作流主流程中（非孤立节点）。
错误信息：`需要连接到其他节点`

## 4. 变量引用校验（跨节点）

### 变量引用格式

```yaml
value_selector:
  - node_id       # 引用的节点 ID
  - output_name   # 引用该节点的某个输出变量名
```

### 校验逻辑

对于每个节点使用的变量引用 `[node_id, output_name, ...]`：

1. **跳过特殊变量前缀**：`sys`, `env`, `conversation`, `rag` — 这些始终有效
2. **检查节点存在性**：`node_id` 对应的节点必须存在
3. **检查输出存在性**：被引用节点必须有名为 `output_name` 的输出变量
4. **检查可达性**：被引用节点必须在当前节点的上游（通过边可达）

### 变量提取方式（各节点不同）

| 节点类型 | 变量来源字段 |
|---------|------------|
| end | `outputs[].value_selector` |
| answer | `answer` 中的模板引用 `{{#node_id.var#}}` |
| llm | `prompt_template[].text` 模板引用 + `context.variable_selector` |
| code | `variables[].value_selector` |
| template-transform | `variables[].value_selector` |
| if-else | `cases[].conditions[].variable_selector` |
| question-classifier | `query_variable_selector` + `instruction` 模板引用 |
| parameter-extractor | `query` + `instruction` 模板引用 |
| knowledge-retrieval | `query_variable_selector` + `query_attachment_selector` |
| http-request | `url` + `headers` + `params` + `body.data` 中的模板引用 |
| tool | `tool_parameters` 中 type=variable 的值 |
| iteration | `iterator_selector` |
| loop | `break_conditions[].variable_selector` |
| variable-assigner | `variables` (直接引用列表) |
| list-operator | `variable` |
| document-extractor | `variable_selector` |
| human-input | `form_content` 中的模板引用 |

### 特殊变量（始终有效）

```
sys.*           — 系统变量 (query, files, user_id 等)
env.*           — 环境变量
conversation.*  — 会话变量
rag.*           — RAG 管道变量
```

## 5. 节点可用输出变量

一个节点能提供什么输出取决于节点类型：

| 节点类型 | 输出变量 |
|---------|---------|
| start | 定义的 `variables` 列表中的变量名 |
| llm | `text` (固定输出) |
| code | `outputs` dict 中定义的 key |
| if-else | 无固定输出（分支路由） |
| knowledge-retrieval | `result` (检索结果) |
| question-classifier | 无固定输出（分类路由） |
| http-request | `body`, `status_code`, `headers` |
| tool | 动态（取决于工具定义） |
| human-input | `form_data` + `user_actions` 中定义的输出 |
| parameter-extractor | 定义的 `parameters` 列表 |
| iteration | `output` |
| template-transform | `output` |
| variable-assigner | 无（赋值操作） |

## 6. 本工具实现方案

### 新增模块：`checklist_validator.py`

实现前端检查清单的 Python 等价版本：

1. **节点配置完整性检查** — 对应前端 `checkValid()`
2. **变量引用合法性检查** — 检查所有 `value_selector` 引用指向有效节点和输出
3. **连接合法性检查** — 已有（`_validate_connectivity()`）

### 集成方式

- `validate` 命令默认包含检查清单检查
- `--strict` 模式下 warning 提升为 error
- 错误信息与 Dify 前端一致
